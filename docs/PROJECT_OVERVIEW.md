# Rubikcubeapp Architecture Overview

## 1. Introduction & Tech Stack

Rubikcubeapp is a React 19 experience that renders and animates a fully interactive 3×3 Rubik’s Cube using `@react-three/fiber` and Three.js. The app focuses on physically accurate cube state management, smooth face rotations, and approachable controls for both keyboard and UI-driven play.

| Layer | Purpose | Key Libraries |
| --- | --- | --- |
| UI | Responsive control overlay, shuffle button, legend | React 19, CSS-in-JS inline styles |
| 3D Rendering | Declarative scene graph, lighting, orbit navigation | `@react-three/fiber`, `@react-three/drei`, Three.js |
| State & Logic | Cubie positions, face colors, rotation math, move queue | Custom utilities in `src/utils/cubeState.ts`, `cubeGeometry.ts` |
| Tooling | DX, bundling, testing | Vite 7, Vitest 4, ESLint 9, TypeScript 5.9 |

## 2. High-Level Architecture Diagram

Textual diagram of the runtime data flow:

```
Keyboard/Shuffle Button
        │
        ▼
   Move Queue (ref) ─────────────┐
        │                        │
        ▼                        │
 RubiksCubeScene state ──┐       │
        │                │       │
        ▼                ▼       │
  <RubiksCube/>   createInitialState/applyMove
        │                │
        ▼                │
 Three.js Mesh graph ◀───┘
        │
        ▼
 Canvas rendering loop (useFrame) → OrbitControls camera interaction
```

* `RubiksCubeScene.tsx` orchestrates user inputs, the cube state array, and the move queue.
* `RubiksCube.tsx` consumes that state to create Mesh instances, attaches affected cubies to a transient animation group, animates rotations, and detaches them back onto the main group to keep transforms aligned.
* Utility modules are pure and reusable; rendering is side-effectful but localized to the component tree.

## 3. Cube State Management

Implementation is centered in `src/utils/cubeState.ts` and `src/utils/cubeGeometry.ts`.

### 3.1 Data Model

```ts
interface Cubie {
  id: string;                 // `cubie-x-y-z`
  position: [number, number, number]; // integer grid
  faceColors: { right?: string; left?: string; ... };
}
```

`CUBIE_POSITIONS` enumerates all 27 positions; `FACE_COLORS` maps canonical face stickers to brand colors.

### 3.2 createInitialState()

* Maps `CUBIE_POSITIONS` to `Cubie` objects.
* Assigns face colors depending on whether a cubie sits on the ±1 plane for each axis.
* Serves both runtime initialization and tests; deterministic order is important for equality checks.

### 3.3 Move Application

* `FACE_AXIS` selects cubies to rotate (axis index plus ±1 coordinate value).
* `ROTATION_TRANSFORMS` encodes 90° rotations as simple `[x,y,z] -> [...]` lambdas per face/direction.
* `FACE_PERMUTATION` semantic table defines sticker reassignment: “dest face receives color from src face.”

```ts
const permutation = FACE_PERMUTATION[face][direction];
newFaceColors[dest] = cubie.faceColors[ permutation[dest] ?? dest ];
```

* `applyMove` clones the array, only transforms cubies at the selected axis plane, and reuses IDs to keep React keys stable.
* `KEY_MAP` translates lowercase keyboard shortcuts to move descriptors; Shift toggles CCW.

## 4. Three.js Rendering & Animation Pipeline

### 4.1 Scene Setup

* `RubiksCubeScene.tsx` mounts a `Canvas` with ambient and directional lights plus `OrbitControls` for mouse/touch navigation. Zoom is enabled; pan is disabled to keep focus on the cube.

### 4.2 Mesh Construction

* `RubiksCube.tsx` memoizes a `THREE.BoxGeometry` sized by `CUBIE_SIZE` ≈ 0.98 to maintain visible gaps.
* Each cubie meshes with cached `MeshStandardMaterial` arrays keyed by cubie ID; materials are disposed when cubies disappear to avoid GPU leaks.

### 4.3 Animation Cycle

1. `moveQueueRef` stores pending moves generated by keyboard or shuffle.
2. `useFrame` checks the queue; when idle it dequeues the next move, calls `selectFace` to determine affected IDs, and attaches their meshes to a dedicated `THREE.Group` (`animGroupRef`).
3. Rotation axis/angle are derived from `getRotationAxis` and `getRotationAngle` using consistent sign conventions (`ANGLE_SIGN`).
4. The animation progresses over `ANIMATION_DURATION` (0.3s) with `easeInOutCubic` easing for smooth acceleration.
5. Once a rotation completes, affected meshes are reattached to the parent group. **Gotcha:** `group.attach(mesh)` bakes world transforms, so the code immediately resets `mesh.rotation` to zero and snaps `mesh.position` to integer grid coordinates. Without this step, cumulative floating-point errors would tilt stickers and break alignment.
6. `onMoveDone` notifies `RubiksCubeScene`, which applies the logical move to the cubie state so the next render reflects updated positions/colors.

### 4.4 Lighting & Materials

* Internal (non-visible) faces use `#333333` to imitate hidden plastic.
* `MeshStandardMaterial` + ambient/directional light produce soft shading without post-processing.

## 5. Input & Controls

### 5.1 Keyboard

* `window.addEventListener('keydown')` listens globally.
* Keys `r/l/u/d/f/b` enqueue clockwise moves; holding Shift flips direction to CCW.
* Input is ignored when a key is unmapped; queue accepts back-to-back moves on different faces but prevents duplicates from the shuffle logic (see below).

### 5.2 Shuffle Button

* Generates 10–50 pseudo-random moves with `generateShuffleMoves()` ensuring no consecutive moves target the same face.
* Disabled while animations run or moves remain queued to avoid overlapping sequences.

### 5.3 Legend Overlay

* `ControlsLegend` renders a blurred glassmorphic card explaining each face, shortcut, and hint to drag for orbit.
* Displays both lowercase key and “⇧Key” to indicate reverse direction.

## 6. Testing Strategy

### 6.1 Unit Tests (`src/utils/__tests__/cubeState.test.ts`)

* Cover `createInitialState` for correct cubie count and sticker placement.
* Verify group-theoretic invariants: applying the same face 4× returns to solved state; CW followed by CCW cancels; known Rubik algorithms (e.g., `(R U R' U')×6`) yield identity.
* Check positional transforms for specific cubies to guard against matrix mistakes.
* Ensure each move keeps 27 unique positions.

### 6.2 Build/Test Pipeline

* `npm run build` → Vite builds the React app.
* `npx vitest run` (or `npm test`) executes the suite in a Node environment per `vite.config.js` (tests match `src/**/*.test.ts(x)`).
* CI recommendation: run lint, build, and vitest sequentially; unit tests are deterministic with no WebGL dependency since logic is pure.

## 7. Extensibility Guidelines

### 7.1 Adding New Moves or Notation

1. Extend `KEY_MAP` with desired shortcuts (e.g., double moves `r2`).
2. Update `FACE_AXIS`, `ROTATION_TRANSFORMS`, and `FACE_PERMUTATION` only if supporting wide/slice moves; keep semantics “dest face gets src color.”
3. Add unit tests mirroring new sequences to lock behavior.

### 7.2 UI Enhancements

* Control Panel: expose buttons for each move; reuse `moveQueueRef` to push descriptors.
* Solvers/Animations: store sequences in React state and stream them into the queue; consider progress indicators leveraging `onAnimatingChange`.
* Themes: factor inline styles into CSS or Tailwind (not currently present) if multiple skins are needed.

### 7.3 Performance Considerations

* Keep `moveQueueRef` mutations synchronous; avoid async operations inside `useFrame` to prevent dropped frames.
* When adding particle effects or post-processing, wrap them inside `<Canvas>` but ensure they do not mutate cubie meshes directly.

## 8. Known Gotchas & Lessons Learned

1. **`group.attach()` resets local transforms:** After rotating a subset of cubies inside `animGroupRef`, each mesh’s world rotation is baked when reattached. The implementation must zero out `mesh.rotation` and snap positions to integers to prevent cumulative drift.
2. **FACE_PERMUTATION semantics:** The permutation table expresses `dest ← src`. Misreading it as “src becomes dest” results in swapped stickers. Always assign `newFaceColors[dest] = oldColors[src]` and default to identity when a face is absent.
3. **Animation gating:** While a move animates, new moves should only enqueue, never start. `isAnimating` state ensures the shuffle button disables, and `onAnimatingChange` toggles UI states accordingly.
4. **Material lifecycle:** Cached `MeshStandardMaterial`s must be disposed when cubies disappear; otherwise GPU memory leaks appear after repeated shuffles.
5. **Keyboard focus:** Global key listener assumes the canvas container is active. For embeddable contexts, gate input handling behind focus checks or provide UI buttons.
